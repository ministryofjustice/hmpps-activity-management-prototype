Discovery opportunities and assumptions

Based on the analysis of the pain points, needs, and problems at Discovery, we’ve identified four core opportunity areas across the journey, Prioritisation & control being one of them:

To enable more requesters in prison to become creators by ensuring that the prioritisation of events reflects the operational goals of the prison and prisoner choice.

A number of key possibilities were also highlighted to explore:

  • Provide an approval mechanism for the activity hub to enable further delegation of the ‘creator’ status across the prison service without jeopardising their management of prisoner time, decreasing the burden on activity staffs that create the majority of appointments and easing bottlenecks
  • Create event ‘bands’ that allow prisoner choice within a given band but allows greater prison control over prioritisation
  • Enable prisons to set the prioritisation of different events against one another to minimise scheduling conflicts (a more technically complex intervention).

Design thinking of initial flows

Three variants were created, each based upon a schedule approach over a calendar approach.

There were a number of reasons for taking this approach:

  • Accessibility - A calendar would prove challenging from an accessibility point of view
  • Scanability - It was assumed that schedules would be easy to scan, as long as dates and information was clear to the user
  • Scalablity - A scannable schedule would be easy to scale and utilising vertical scroll, would be easy to present a lot of information useful to each userbase.

Version 1 - Including who created the appointment and visualising clash information

For this version, we introduced:

  • The creator of the appointment
  • Highlighted where clashes occurred

At this stage, no design thinking was placed around showing priority of appointments for this version, instead focussing on minimal additional information to see gains.

Version 2 - Further iterated, introducing banded priorities

For this version, we introduced:

  • The concept of placing a priority around the type of appointment being created and wether the appointment being created was of a higher or lower priority

Our assumption here was that users would understand and find the idea of priorities useful when compared to the session they are creating themselves, allowing the user to decide whether to set an appointment at the same time if a higher or lower priority appointment exists for that time already.

Version 3 - Inclusion of numeric priorities

For this version, we introduced:

  • Swapping the idea of a 'Higher' 'Lower' priority set for a numeric one.

User research - What we learnt

Insights around the current process of creating or requesting an appointment

  • Appointment clashes are not a universal issue across prisons but can have significant impact when they are
  • Users already have a clear understanding of event priority but this can change with the context of an event
  • Visibility and awareness of the schedule is the key ingredients for avoiding appointment clashes
  • Most users want to view a schedule further into the future than the current two-week view

Prototype feedback

Version 1 - Clash and created by information

  • Each participant saw a benefit in displaying where clashes occurred and this was immediately obvious to all participants. Each participant used the information around clashes and decided that they would not create an appointment around that date and time
  • Displaying the name of the creator of an appointment was also seen as a benefit, although, some participants stated that they would rather see the appointment host as well as the creator

    • {% from "govuk/components/warning-text/macro.njk" import govukWarningText %} {{ govukWarningText({ text: "Watchpoints around version 1", iconFallbackText: "Warning" }) }}
  • While overall the feedback supported the use of version 1, it was noted that this version does not lend itself to users who are new to the sector or role that they are covering and instead relies on internal 'knowledge' surrounding the various levels of priority.
  • One participant did flag a potential security risk with displaying any name, especially if they were to appear on the unlock list

User research recommendations for version 1

We should consider including the name of the appointment host within this view but hiding any names when it comes to printing lists which could be viewed by prisoners.



Version 2 - Schedule with priority tags

  • The use of these were immediately obvious to users and clearly implied a priority in proportion to the appointment being set up.

    • {% from "govuk/components/warning-text/macro.njk" import govukWarningText %} {{ govukWarningText({ text: "Watchpoints around version 2", iconFallbackText: "Warning" }) }}
  • The priority of events ‘will be ignored’ unless it accounts for the nuance that exists within prisons related to the context of an event - e.g. a chaplain appointment to discuss a bereavement would often be deemed higher priority over most other scheduled events

User research recommendations for version 2

We need to carefully consider how nuanced the priority tags would need to be and whether there would be unintended consequences as a result
